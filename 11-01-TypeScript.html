    TypeScript

    # TypeScript优缺点

    优点

    • JS是运行中抛出错误，TS是在编辑器中告知，提升开发效率，不用花大量时间去写单测，同时避免大量时间排查bug
    • 前后端联调要去看接口文档上的字段类型，TS会帮助识别当前类型（类型推倒）
    • 友好的在编辑器里提示错误，避免代码运行时类型隐式转换踩坑

    缺点

    • 可能和一些插件库结合的不是很完美

    # TS && JS代码运行流程

    JS运行流程如下，依赖NodeJs环境和浏览器环境

    • 将JavaScript代码转换为JavaScript-AST
    • 将AST代码转换为字节码
    • 运算时计算字节码

    TS运行流程，三步执行完继续同上操作，让浏览器解析

    • 将TS代码编译为 TS-AST
    • 检查AST代码上类型检查
    • 类型检查后，编译为JS代码
    • JS代码转换为JS-AST
    • 将AST代码转换为字节码
    • 运算时计算字节码

    # TS和JS区别

    类型系统特性 JavaScript TypeScript
    类型是如何绑定？ 动态 静态
    是否存在类型隐式转换？ 是 否
    何时检查类型？ 运行时 编译时
    何时报告错误 运行时 编译时
    • 类型绑定：如果该变量没有定义类型，那么TypeScript会自动类型推导出来。
    • 类型转换：TS中1+true会报错，提示number类型不能和boolean类型进行运算
    • 类型检查：JS只有在程序运行时才能检查类型。类型也会存在隐式转换，很坑。

    # TS总共围绕两种模式展开

    显式注解类型——声明时带上注解：

    let name: string = "前端娱乐圈";
let age: number = 38;
let hobby: string[] = ["write code", "玩游戏"]

    推导类型——推导类型就是去掉显示注解，系统自动会识别当前值是一个什么类型的。

    npm i -g typescript
npm i ts-node

    三步：

    • index.ts

    # 基础知识

    # 基础静态类型

    • number
    • string
    • boolean
    • null
    • undefined
    • void——代表无效的，一般只用在函数上，告诉别人这个函数没有返回值
    function fn(): void {} // 正确

function testFn(): void {
    return 1; // 报错，不接受返回值存在
}

function fn1(): void { return undefined} // 显示返回undefined类型，也是可以的

function fn2(): void { return null} // 显示返回null类型也可以，因为 null == undefined
    • never
    const test: never = null; // 错误
const test1: never = undefined // 错误

function Person(): never { // 正确，因为死循环了，一直执行不完
    while(true) {}
}

function Person(): never { // 正确，因为递归，永远没有出口
    Person()
}

function Person(): never { // 正确 代码报错了，执行不下去
    throw new Error()
}
    • any——遇事不决就用any！
    • unknown——unknown相当于是any类型，区别：
    let valueAny: any = "";
let valueUnknown: unknown = "";

valueAny = "蛙人";
valueUnknown = "前端娱乐圈";

let status: null = false;
status = valueAny; // 正确
status = valueUnknown // 报错，不能将unknown类型分配给null类型

    别人不接受unknown类型，而unknown类型接受别人

    function testFnQ(a:number, b:number) {
    return a + b

    # 枚举Enum

    Enum枚举类型，可以设置默认值，如果不设置则为索引。

    enum color {
    RED,
    BLUE = "blue",
    GREEN = "green"
}

// color["RED"] 0
// color["BLUE"] blue

// 报错
enum color {
    RED,
    BLUE = "blue",
    GREEN
}

// good
enum color {
    RED,	   // 0
    BLUE = 4,  // 4
    GREEN      // 5
}

    enum枚举类型还可以反差，通过value查key值。像我们json对象就是不支持这种写法的。

    enum color {
    RED,	   // 0
    BLUE = 4,  // 4
    GREEN      // 5
}

console.log(color[4]) // BLUE
console.log(color[0]) // RED

    # 接口Interface

    • 接口怎么复用

    接口interface就是方便我们定义一处代码，多处复用

    可以看到使用interface关键字定义一个接口，然后赋值给这对象的多个变量，实现复用。——主要就是请求后端接口时常需要对传输参数和接收参数做接口定义。

    • readonly修饰符——只可读状态，不可更改。
    interface Types {
    readonly name: string, 
    readonly age: number
}

const testObj: Types = { name: "前端娱乐圈", age: 18 }

const testObj1: Types = { name: "蛙人", age: 18 }

testObj.name = "张三" // 无法更改name属性，因为它是只读属性
testObj1.name = "李四" // 无法更改name属性，因为它是只读属性
    • ?可选修饰符——定义该接口对象时，该属性时选填的（后端接口定义有些参数是选填的）
    • extends继承——跟ES6Class类一样，使用extends关键字。
    interface Types {
    readonly name: string, 
    readonly age: number,
    sex?: string
}

interface ChildrenType extends Types { // 这ChildrenType接口就已经继承了父级Types接口
    hobby: []
}
    
const testObj: ChildrenType = { name: "前端娱乐圈", age: 18， hobby: ["code", "羽毛球"] }
    • propName扩展——可以写入不在interface里面的属性。
    interface Types {
    readonly name: string, 
    readonly age: number,
    sex?: string,
    [propName: string]: any // propName字段必须是 string类型 or number类型。 值是any类型，也就是任意的
}

const testObj: Types = { name: "前端娱乐圈", age: 19, hobby: [] }
    • Type

    这个是声明类型别名使的，别名类型只能定义是：基础静态类型、对象静态类型、元组、联合类型。

    type别名不可以定义interface。

    type Types = string;

type TypeUnite = string | number

const name: typeUnite = "前端娱乐圈"
const age: typeUnite = 18

    那么type类型别名和interface接口有什么区别呢?

    答案：

    1. 第一个Types类型别名type不允许出现重复名字，interface接口可以出现重复类型名称，如果重复出现则是，合并起来也就是变 { name：string, age: number }，同时后续声明的接口，必须跟前面声明的同名属性类型必须保持一致。

    2. type支持表达式 interface不支持——type testType = typeof count

    3. type 支持类型映射，interface不支持

    type keys = "name" | "age"  
type KeysObj = {
    [propName in keys]: string
}

const PersonObj: KeysObj = { // 正常运行
    name: "蛙人",
    age: "18"
} 

interface testType {
    [propName in keys]: string // 报错
}
    • 联合类型
    const statusTest: string | number = "前端娱乐圈"

const flag: boolean | number = true

function testStatusFn(params: number | string) {
    console.log(params.toFixed()) // 报错
}

testStatusFn(1)

    函数参数类型不能类型自动推导，更何况现在用上联合类型，系统更懵逼了，不能识别当前实参的类型。所以访问当前类型上的方法报错——就有了类型保护

    类型保护：

    1. typeof
    // 正常
function testStatusFn(params: string | number) {
    if (typeof params == "string") {
        console.log(params.split)
    }

    if (typeof params == "number") {
        console.log(params.toFixed)
    }
}

testStatusFn(1)
    1. in
    2. as 断言
    // 正常
function testStatusFn(params: frontEnd | backEnd) {
    if ("name" in params) {
        const res = (params as frontEnd).name
        console.log(res)
    }
    
    
    if ("age" in params) {
        const res = (params as backEnd).age
        console.log(res)
    }
}

testStatusFn({age: 118})
    • 交叉类型——交叉类型就是跟联合类型相反，它用&表示，交叉类型就是两个类型必须存在。&& 两个接口类型中都出现了同名属性，但是类型不一样，这时类型就会变为never。
    • 泛型
    function test(a: string | number, b: string | number) {
    console.log(a, b)
}
test(1, "前端娱乐圈")

    函数参数注解类型定义string和number

    有个需求，就是实参我们必须传入同样的类型。 这时就需要用到泛型了，泛型是专门针对不确定的类型使用。泛型的使用大部分都是使用<T>，当然也可以随便使用，如：<Test>、<Custom>都可以。

    function test<T>(a: T, b: T) {
    console.log(a, b)
}
test<number>(1, "前端娱乐圈") // 调用后面跟着尖括号这就是泛型的类型，这时报错，因为在调用的使用类型是number，所以只能传入相同类型的

test<boolean>(true, false) 

test<string>("前端娱乐圈", "蛙人")
test<any>(1, "前端娱乐圈")// 可以传入不同的类型

    但是上面这种又有一种问题，它可以传入对象，但是如果我们只希望传入number类型和string类型。那么我们泛型也给我们提供了约束类型。泛型使用extends进行了类型约束，只能选择string、number类型。

    function test<T extends number | string, Y extends number | string>(a: T, b: Y) {
    console.log(a, b)
}

test<number, string>(18, "前端娱乐圈")

test<string, number>("前端娱乐圈", 18)

    这时，传入泛型时使用，逗号分隔，来定义每一个类型希望是什么。记住，只有我们不确定的类型，可以使用泛型。
